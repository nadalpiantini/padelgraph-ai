## D4 — Detection backbone: YOLOX (NOT YOLOv8/Ultralytics)

- **valid_from:** 2026-05-28
- **transaction_time:** 2026-05-28
- **context:** YOLOv8 (Ultralytics, 57K★) is the dominant detection model in 2026, but it is licensed AGPL-3.0. AGPL contaminates any code that uses it — any derivative must also be AGPL, which kills commercial SaaS use cases.
- **alternatives considered:**
  - YOLOX (Megvii, Apache 2.0, 9K★) — slightly less mature ecosystem but commercial-OK
  - YOLOv9 — AGPL, fragmented forks
  - RT-DETR (Baidu, Apache 2.0) — transformer-based, requires PaddlePaddle ecosystem (overhead)
  - Detectron2 (Meta, Apache 2.0) — heavier framework, more general-purpose
- **decision:** YOLOX as default detection backbone for ball + players.
- **rationale:** Apache 2.0 license is non-negotiable for a project that may become commercial SaaS later. Verified via Context7 that YOLOX install + usage is stable in Q2 2026.

## D5 — Multi-camera fusion library: Kornia (NOT PyTorch3D or hand-rolled OpenCV)

- **valid_from:** 2026-05-28
- **transaction_time:** 2026-05-28
- **context:** Phase 1 sports CV landscape audit (2026-05-28) recommended Kornia (10K★ Apache, built by OpenCV authors) for multi-camera triangulation. Alternative was PyTorch3D (overkill) or hand-rolled OpenCV (more code, slower iteration).
- **alternatives considered:** PyTorch3D, custom OpenCV `cv2.triangulatePoints`, building from scratch
- **decision:** Kornia for triangulation 3D + camera calibration helpers.
- **rationale:** Purpose-built differentiable computer vision, reduces custom code ~80%, Apache 2.0, used in production by major CV projects.

## D8 — License: Apache 2.0 (NOT MIT, NOT BSD, NOT GPL)

- **valid_from:** 2026-05-28
- **transaction_time:** 2026-05-28
- **context:** Repo needs a permissive license that allows commercial reuse + handles patent grants explicitly.
- **alternatives considered:** MIT (too minimal — no patent grant), BSD-3 (similar to MIT), MPL-2.0 (weak copyleft), GPL/AGPL (viral — rejected)

- **rationale:** Industry standard for computer vision libraries, explicit patent grant matters for a domain where patents are real risk, compatible with all major OSS licenses for downstream consumers.

## D12 — Business panel synthesis: reframe from "product" to "schema + dataset + reputation play"

- **valid_from:** 2026-05-28
- **transaction_time:** 2026-05-28
- **context:** Post-Epic-1-shipped audit invoked Sephirot `business-panel-experts` agent in debate mode (Christensen / Porter / Kim·Mauborgne / Taleb / Drucker). Full debate captured verbatim in `BUSINESS_PANEL.md`.
- **3 weakest assumptions surfaced:**
  1. "OSS reference implementation is a defensible position" — it is NOT. Apache 2.0 with no hosted/dataset/enterprise layer gives the downstream market away.
  2. "Veo's absence = market opening" — Veo's absence reflects padel's installed-base economics, NOT opportunity. Real incumbents are Pixellot, PlaySight, phone+mount stack. Buyer (clubs) has near-zero analytics budget.
  3. "Business model TBD is fine for now" — Apache 2.0 was chosen BEFORE the model. That ordering forecloses options (dual-license, proprietary dataset, hosted enterprise) that cannot be reopened.
- **3 most likely failure modes:**
  1. Zombie repo — 12 months in, <200 stars, no contributors, Epic 2 never ships because capacity goes to Empleaido. D24 fails.
  2. Successful but extracted — a hardware bundler (BLAUBECK, Pixellot reseller) wraps padelgraph-ai, sells the bundle, captures margin. Alan gets a GitHub thank-you and zero revenue.
  3. Strategic drift — padelgraph-ai consumes Sephirot D2 "1 project active" slot that Empleaido (closer to revenue) needs for push-mode. Opportunity cost compounds.
- **1 missing path forward (panel consensus):** Stop calling it a product. Reframe as 12-month standards + dataset + reputation play. Keep Apache 2.0 — it's correct for THIS purpose — but spend real effort on:
  - (a) define MatchAnalysis JSON as the open schema for padel data,
  - (b) sign 3-5 LATAM clubs/federations as design partners contributing calibration profiles + annotated footage to a *separately licensed* dataset (CC-BY-NC or proprietary),
  - (c) position Alan as the LATAM padel data infrastructure person — feeds Filmatron sports-cinema + creates inbound consulting.
  - Rationale: code depreciates, schema+dataset+relationships appreciate. Founder built the wrong asset first; reverse priority while code investment is still fresh.
- **decision:** Adopt the reframe. Next 60 days: focus on schema canonicalization + design-partner outreach (3-5 LATAM clubs). Code work continues but as bait, not as the asset. Update PRD.md success criteria mes 6 to include "≥1 LATAM federation or club signed as design partner contributing calibration profile" as criterion #9 (extends original 8).
- **verdict (panel one-liner):** "Needs strategic reframe within 60 days — keep shipping code, but treat it as bait for the dataset+schema+reputation play, not as a product. Otherwise: pivot or retire by month 6, not month 12."
- **rationale:** Five experts converged from independent angles to the same diagnosis. When that happens, the founder should listen. The reframe is cheap (does not require rewriting code) but high-leverage (changes what success means + what to measure). Captured here so future Sephirot sessions reference it before scoping Epic 2 capacity.
